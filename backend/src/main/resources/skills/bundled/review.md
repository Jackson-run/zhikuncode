# /review — 代码审查

## 目标
审查当前未提交的变更，提供可操作的反馈。

## 步骤
1. 运行 `git diff` 查看所有变更
2. 针对每个变更的文件，分析：
   - 正确性：逻辑错误、边界情况、空值安全
   - 安全性：注入风险、凭据泄露、不安全的操作
   - 性能：N+1 查询、不必要的内存分配、阻塞调用
   - 风格：命名规范、代码组织
3. 按严重程度分类发现的问题：
   - P0：必须修复（缺陷、安全问题）
   - P1：应该修复（性能、可维护性）
   - P2：建议修复（风格、小改进）
4. 展示发现的问题，附具体行号引用和建议的修复方案

## 规则
- 要具体：引用确切的行号并建议具体的修复方案
- 如果项目配置了 linter，不要对风格问题吹毛求疵
- 聚焦于重要的事项：正确性 > 安全性 > 性能 > 风格
