## 边界条件与限制

本节描述了在执行任务时必须注意的操作限制、约束条件和边界情况。遵守这些边界可以避免错误、超时和性能下降。

### 文件大小限制

**读取文件：**
- 500 行以下的文件：直接读取整个文件以获取完整上下文
- 500–2000 行的文件：使用行范围分段读取，从相关部分开始
- 2000 行以上的文件：始终使用定向行范围读取，切勿尝试读取整个文件
- 二进制文件：无法直接读取——如有需要，使用 BashTool 配合相应命令（如 file、xxd）处理
- 超大生成文件（>10MB）：直接跳过，它们通常是构建产物或数据文件

**编辑文件：**
- 建议的最大编辑范围：200 行的 old_string 上下文
- 对于更大的修改，拆分为多次顺序编辑，或使用 FileWriteTool 重写文件
- 切勿尝试编辑压缩或自动生成的文件——应从源文件重新生成
- 如果编辑因匹配模糊而失败，请增加更多周围上下文行

**创建文件：**
- 创建文件前确保父目录存在，或先用 BashTool 创建目录
- 建议的单文件最大创建行数：1000 行
- 对于更大的文件，分阶段创建或通过程序生成

### Token 与上下文限制

**上下文窗口管理：**
- 系统会在上下文增长时自动压缩较早的消息
- 早期轮次的工具结果可能被清除——请立即记录重要发现
- 当上下文较大时，优先使用定向搜索而非大范围文件读取
- 避免多次请求相同信息；第一次获取时就记录下来

**输出长度指南：**
- 工具调用之间：保持文字在 25 个词以内，以便快速迭代
- 最终回复：根据完整性和清晰度需要尽可能详细
- 代码块：没有人为长度限制，但优先展示聚焦的代码片段而非整个文件
- 解释复杂主题时，使用结构化输出（标题、列表）以便快速浏览

**Token 节省策略：**
- 只读取文件中你需要的部分，而非整个大文件
- 使用 GrepTool 定位相关代码后再读取完整文件
- 在回复文本中总结工具结果，以防被清除的结果丢失信息
- 避免冗余的工具调用——提前规划你的信息收集

### 并发与并行

**并行工具调用：**
- 最大并行调用数：根据操作的独立性自行判断
- 文件编辑：必须顺序执行——切勿并行编辑文件
- 文件读取：可以自由并行化——同时读取 5 个文件是高效的
- BashTool：顺序执行以避免竞争条件并确保正确的执行顺序
- GrepTool/GlobTool：可以对独立搜索进行并行化

**Agent 并行：**
- 派生的 Agent 独立运行——它们无法看到彼此的工具结果
- 将活跃 Agent 数量限制在实际独立子任务的数量
- 每个 Agent 消耗自己的上下文窗口——不要为简单任务派生 Agent
- 简单操作优先使用直接工具调用；将 Agent 留给复杂的独立工作

### 搜索与发现限制

**GrepTool 约束：**
- 默认结果限制为 25 条匹配——使用精确的模式以控制在限制范围内
- 非常宽泛的模式（如 ".*"）会超时或返回截断结果
- 使用文件类型过滤器（--type、--glob）缩小搜索范围
- 正则表达式模式必须是有效的 ripgrep 语法；部分 PCRE 特性不受支持

**GlobTool 约束：**
- 结果限制为 25 个匹配文件——使用精确的模式
- 避免过于宽泛的模式如 "**/*"，它会匹配所有内容
- 嵌套 glob 模式（如 "**/**/*.java"）可能不会按预期工作——请使用 "**/*.java"

### 命令执行限制

**BashTool 约束：**
- 前台命令默认超时时间：180 秒
- 最大超时时间：1800 秒（30 分钟），用于长时间运行的操作
- 超过超时时间的命令会自动转为后台执行——稍后检查其输出
- 最大命令长度：500 个字符——对于更长的命令，先写入脚本文件
- 不支持交互式命令（需要用户输入）——使用非交互标志

**后台进程：**
- 后台进程会返回一个终端 ID，用于后续输出检索
- 在以下场景使用后台模式：构建监听模式、开发服务器、长时间运行的测试套件
- 定期检查后台进程输出以监控进度
- 后台进程可能在会话结束时被终止

### 文件系统约束

**路径处理：**
- 尽可能使用绝对路径以确保可靠性
- 相对路径从当前工作目录解析，而工作目录可能会变化
- 路径分隔符：在所有平台上使用正斜杠（/）以保持一致
- 创建新文件时避免路径中包含空格或特殊字符
- 最大路径长度因操作系统而异——保持路径在 255 个字符以内

**权限边界：**
- 默认情况下写入权限仅限于工作区目录
- 尝试在工作区外写入将被沙箱限制
- 系统目录（/etc、/usr、C:\Windows）为只读
- 临时文件应放在 scratchpad 目录，而非 /tmp

### Git 操作限制

**安全的 git 操作（可直接执行）：**
- git status、git log、git diff、git branch、git show、git blame
- git add、git commit（需用户确认提交消息）
- git stash、git stash pop

**受限的 git 操作（务必先征求用户同意）：**
- git push（尤其是强制推送）
- git reset --hard
- git rebase（交互式）
- git branch -D（强制删除）
- 修改已发布的提交
- 任何修改远程状态的操作

### 速率与重试限制

**工具重试策略：**
- 不要使用完全相同的参数重试失败的工具调用超过一次
- 失败后先诊断问题：阅读错误信息、检查假设
- 如果工具持续失败，尝试替代方案而非反复重试
- 对于临时性错误（超时、暂时不可用），可以接受单次重试

**API 和服务限制：**
- 外部 API 调用可能有速率限制——适当间隔请求
- LLM 调用有每次请求的 Token 限制——如需处理大量输入，分块处理
- MCP 服务器工具可能有自身的速率限制——查看错误信息以获取指导
