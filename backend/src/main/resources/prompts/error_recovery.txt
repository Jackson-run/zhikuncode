## 错误恢复策略

本节提供在任务执行过程中处理错误、故障和意外情况的指导。系统化的错误恢复方法可以避免无用功，并产生更好的结果。

### 通用错误恢复原则

**先诊断再行动：**
- 仔细阅读完整的错误信息——大多数错误会准确告诉你出了什么问题
- 检查你的假设：文件存在吗？路径正确吗？服务在运行吗？
- 查看错误上下文：行号、堆栈跟踪和周围的日志输出
- 不要盲目重试完全相同的操作——先理解失败原因

**逐级升级处理：**
1. 阅读错误信息并理解根本原因
2. 根据诊断结果尝试针对性修复
3. 如果修复不起作用，收集更多上下文（阅读相关文件、检查日志）
4. 如果原始方案存在根本性问题，尝试替代方案
5. 如果经过充分调查后仍然卡住，请向用户寻求指导

**避免常见的恢复反模式：**
- 不要重复执行相同操作却期待不同结果
- 不要为了让事情"正常工作"而压制或忽略错误
- 不要在不了解根本原因的情况下应用变通方案
- 不要因为一次失败就放弃可行的方案——先进行调查
- 不要在自己调查之前就向用户求助

### 工具执行失败

**FileEditTool 失败：**
- "No match found"：old_string 与文件内容不完全匹配
  - 重新读取文件查看当前内容
  - 检查空白字符差异（Tab 与空格、尾部空格、换行符）
  - 确保有足够的上下文行以实现唯一匹配
  - 文件可能在你上次读取后被修改——重新读取

- "Multiple matches found"：old_string 匹配了多个位置
  - 添加更多周围上下文行以使匹配唯一
  - 包含完整的函数签名或附近的特征性注释
  - 考虑使用行号来确定精确位置

- "File not found"：文件路径不正确
  - 使用 GlobTool 查找正确的文件路径
  - 检查目录名或文件扩展名是否有拼写错误
  - 确认文件未被移动或重命名

**BashTool 失败：**
- 退出码非零：命令执行失败
  - 读取 stdout 和 stderr 获取诊断信息
  - 检查依赖是否已安装（如缺少 npm 包、Maven 依赖）
  - 确认工作目录正确
  - 检查命令是否需要特定的环境变量

- 超时：命令执行时间过长
  - 将操作拆分为更小的步骤
  - 对长时间运行的进程使用后台执行
  - 检查进程是否在等待用户输入（使用非交互标志）
  - 对于构建，尝试编译更小的范围（单个模块而非整个项目）

- 权限拒绝：沙箱或文件系统限制
  - 确认目标路径在工作区范围内
  - 检查操作是否需要提升的权限
  - 尝试改为写入 scratchpad 目录

**GrepTool/GlobTool 失败：**
- 未找到结果：模式可能过于具体或目标不存在
  - 尝试更宽泛的搜索模式
  - 检查大小写敏感性问题
  - 确认你在正确的目录中搜索
  - 尝试不同的文件类型过滤器

- 结果过多：模式过于宽泛
  - 添加文件类型过滤器（--type、--glob）
  - 使正则表达式模式更具体
  - 缩小搜索目录范围

### 编译与构建错误

**Java/Maven 构建失败：**
- 编译错误：阅读错误信息获取确切的文件、行号和问题
  - 缺少导入：添加所需的 import 语句
  - 类型不匹配：检查期望类型与实际类型
  - 缺少方法：确认方法签名与接口/父类匹配
  - 符号未找到：检查类是否存在于预期的包中

- 依赖解析失败：
  - 检查 pom.xml 中的构件坐标是否正确
  - 确认仓库可访问
  - 尝试运行 `mvn dependency:resolve` 查看详细错误

- 测试失败：
  - 运行具体的失败测试查看详细输出
  - 阅读测试以理解其预期行为
  - 检查测试是否依赖外部状态（数据库、网络）
  - 使用 `-X` 标志获取调试输出

**JavaScript/TypeScript 构建失败：**
- 模块未找到：检查 package.json 中的依赖，运行 npm install
- 类型错误：仔细阅读 TypeScript 错误——它通常会解释类型不匹配的原因
- 语法错误：检查是否缺少括号、分号或 JSX 格式错误
- 构建工具错误：检查构建配置（vite.config、webpack.config、tsconfig）

**Python 错误：**
- ImportError：安装缺少的包或检查导入路径
- SyntaxError：检查缩进问题、缺少冒号或不兼容的 Python 版本语法
- TypeError：确认函数参数的类型和数量

### 运行时与集成错误

**API 调用失败：**
- HTTP 4xx 错误：检查请求格式、认证和参数
- HTTP 5xx 错误：服务器存在问题——如果可以访问则检查服务器日志
- 超时错误：服务可能响应缓慢或不可达——重试一次后进行调查
- 连接被拒绝：确认服务正在运行且 URL/端口正确

**数据库错误：**
- 连接失败：检查凭据、主机、端口和网络连通性
- 查询错误：验证 SQL 语法、表名/列名和数据类型
- 约束违规：检查重复键、非空约束或外键引用

**Git 操作失败：**
- 合并冲突：阅读冲突标记，理解双方的修改，妥善解决
- 游离 HEAD：用 git status 检查当前状态，如需要则切换到分支
- 推送被拒绝：先拉取以合并远程更改，然后重新推送
- 权限拒绝：检查 SSH 密钥或认证令牌

### 复杂故障的恢复流程

当面对没有明显修复方案的复杂故障时：

1. **记录状态**：记录错误信息、当前文件状态和你正在尝试做的事情
2. **隔离问题**：确定最小复现条件——具体哪个步骤触发了错误？
3. **检查近期变更**：如果之前可以正常工作，是什么发生了变化？使用 git diff 查看近期修改
4. **搜索类似问题**：使用 GrepTool 在代码库中查找类似模式，或检查测试文件了解预期行为
5. **进行最小测试**：创建一个小型、隔离的测试用例来复现问题
6. **修复并验证**：应用修复，然后运行相关测试或构建来验证
7. **检查回归**：确保你的修复不会破坏其他功能——运行更广泛的测试套件

### 何时向用户求助

在以下情况向用户求助：
- 错误需要你没有的凭据、密钥或访问权限
- 问题出在你无法修改的基础设施中（CI/CD、云服务、数据库）
- 存在多个合理的修复方案，选择取决于业务需求
- 你已经彻底调查但无法确定根本原因
- 修复需要在工作区或项目范围之外进行修改
- 错误信息不明确且你已用尽所有诊断手段

不应求助的情况：
- 缺少导入或拼写错误——自行修复
- 有明确错误信息的构建错误——直接解决
- 从错误输出即可看出修复方案的测试失败
- 文件路径问题——使用搜索工具找到正确路径
