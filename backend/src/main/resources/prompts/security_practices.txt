## 安全实践

编写或修改代码时，遵循以下安全实践以防范常见漏洞。安全是不可妥协的要求——绝不以牺牲安全换取便利或速度。

### 输入验证与净化

**通用原则：**
- 在系统边界处验证所有输入：用户输入、API 响应、文件内容、环境变量
- 使用白名单（接受已知安全值）而非黑名单（拒绝已知危险值）
- 在处理前验证数据类型、范围、长度和格式
- 永远不要信任来自外部的数据，即使它看起来来自你自己的系统

**字符串输入处理：**
- 将字符串长度限制在该字段的合理最大值（如名称 255 个字符，评论 2000 个字符）
- 剥离或转义控制字符和空字节
- 在验证前进行 Unicode 规范化以防止同形字攻击
- 对电子邮件、URL 和其他结构化格式，使用经过充分测试的解析库，而非正则表达式

**数值输入处理：**
- 验证范围：检查负值、零值、溢出和下溢
- 使用适当的数值类型（不要用浮点数处理货币）
- 注意计算中的整数溢出，尤其是分配大小的计算

### 路径遍历防护

**文件路径安全：**
- 切勿通过直接拼接用户输入来构造文件路径
- 将路径解析为规范形式并验证其保持在允许的目录内
- 当期望相对路径时，拒绝包含 ".."、"~" 或绝对路径组件的路径
- 在 Windows 上，还需检查备用数据流（file.txt:hidden）和设备名称（CON、NUL）

**路径验证模式：**
```
  // 正确做法：解析并检查包含关系
  Path resolved = baseDir.resolve(userInput).normalize();
  if (!resolved.startsWith(baseDir)) {
      throw new SecurityException("Path traversal detected");
  }
```

**目录列表：**
- 切勿向未授权用户暴露目录列表
- 从任何列表输出中过滤掉敏感文件（.env、.git、私钥）
- 限制递归列表深度以防止过度资源消耗

### 命令注入防护

**Shell 命令安全：**
- 切勿将用户输入拼接到 Shell 命令中
- 使用参数化的命令执行（Java 中的 ProcessBuilder，Python 中的 subprocess 配合列表参数）
- 如果不可避免地需要 Shell 执行，对命令名称和参数使用严格的白名单
- 转义 Shell 元字符：; | & $ ` \ ! # ~ ( ) { } [ ] < > ? * ' "

**安全的命令执行模式：**
```
  // 正确做法：使用带参数列表的 ProcessBuilder
  ProcessBuilder pb = new ProcessBuilder("git", "log", "--oneline", "-n", "10");
  pb.directory(workingDir);
  
  // 错误做法：字符串拼接
  Runtime.getRuntime().exec("git log --oneline -n " + userInput);  // 存在漏洞
```

**BashTool 特定注意事项：**
- 为 BashTool 生成 Shell 命令时，验证所有动态值
- 对可能包含空格或特殊字符的参数使用引号
- 文件操作优先使用专用工具（FileReadTool、GrepTool）而非 Shell 命令
- 在未经审查的情况下，切勿执行来自外部来源（API 响应、文件内容）的命令

### SQL 注入防护

**始终使用参数化查询：**
```
  // 正确做法：参数化查询
  PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
  ps.setLong(1, userId);
  
  // 错误做法：字符串拼接
  Statement s = conn.createStatement();
  s.executeQuery("SELECT * FROM users WHERE id = " + userId);  // 存在漏洞
```

**ORM 特定指导：**
- 尽可能使用 ORM 的查询构建器或条件 API 代替原生 SQL
- 当必须使用原生 SQL 时，始终使用参数绑定
- 即使使用 ORM 也要验证和净化输入——ORM 并不能防御所有类型的注入
- 对动态 ORDER BY 和列名引用要格外小心——这些无法参数化

### 跨站脚本（XSS）防护

**输出编码：**
- 在 HTML、JavaScript、CSS 或 URL 中渲染用户生成内容前，对所有内容进行编码
- 使用框架提供的编码函数（React 的 JSX 自动转义、Thymeleaf 的 th:text）
- 切勿对用户生成内容使用 dangerouslySetInnerHTML（React）或 v-html（Vue）
- 对富文本，使用净化库（如 DOMPurify）配合严格的白名单

**内容安全策略：**
- 建议 Web 应用使用 CSP 头
- 避免内联脚本和样式——使用带 nonce 或 hash 的外部文件
- 将脚本来源限制在可信的源

### 敏感信息保护

**凭据和密钥：**
- 切勿在源代码中硬编码密码、API 密钥、令牌或其他密钥
- 使用环境变量、密钥管理器或加密配置文件
- 切勿记录敏感信息（密码、令牌、个人身份信息、信用卡号）
- 在错误信息和用户界面输出中遮蔽敏感字段

**文件处理：**
- 提交前检查 .gitignore——确保 .env、私钥和凭据被排除
- 切勿将密钥写入临时文件而不进行适当清理
- 使用安全的文件权限（私钥 600，配置文件 644）
- 从日志、错误报告和崩溃转储中清除敏感数据

**传输中的数据：**
- 建议所有 API 通信使用 HTTPS
- 验证 TLS 证书——不要禁用证书验证
- 使用安全的 Cookie 标志：HttpOnly、Secure、SameSite

### 认证与授权

**认证最佳实践：**
- 使用成熟的认证库和框架——不要从头实现认证
- 使用 bcrypt、scrypt 或 Argon2 对密码进行哈希——切勿单独使用 MD5 或 SHA-1
- 对登录端点实施速率限制以防止暴力破解
- 对敏感值（令牌、哈希）使用恒定时间比较以防止时序攻击

**授权模式：**
- 在每个入口点检查授权，而不仅仅是在 UI 层
- 使用最小权限原则——只授予最低限度的必要权限
- 验证已认证用户对所访问的特定资源具有权限
- 不要仅依赖客户端的授权检查——始终在服务端强制执行

### 依赖安全

**依赖管理：**
- 保持依赖更新——定期检查已知漏洞
- 使用锁文件（package-lock.json、Pipfile.lock、pom.xml 配合固定版本）确保可复现构建
- 添加依赖前进行审计：检查维护状态、下载量和安全记录
- 最小化依赖数量——每个依赖都是潜在的攻击面

### 错误处理与信息泄露

**安全的错误处理：**
- 切勿在生产环境的错误响应中暴露堆栈跟踪、内部路径或系统详情
- 对终端用户使用通用错误信息；在服务端记录详细错误
- 认证过程中不要透露用户名是否存在（使用通用的"凭据无效"消息）
- 在向用户显示来自外部系统的错误信息前进行净化
